# 3. 리액티브 프로그래밍이란?
- 리액티브 스트림 구현체 Flux, Mono를 사용하여 발생하는 데이터를 바로바로 리액티브하게 처리
- 비동기로 동작 - 논 블로킹하게 동작 해야한다.

- 리액티브 프로그래밍 필수 요소 
  - 데이터가 준비 될 때 마다 바로바로 리액티브하게 처리 -> 리액티브 스트림 구현체 Flux, Mono를 사용하여 발생하는 데이터 바로바로 처리
  - 로직을 짤 때는 반드시 논 블로킹하게 짜야함 -> 이를 위해 비동기 프로그래밍이 필요

# 4. 스레드란?
- 하나의 작업 흐름을 실행시켜 주는 것
- 비동기 프로그래밍을 위해 필요한 기본 요소
- Thread = 실
  - 유저 요청 -> 스레드는 cpu에 자원 할당 받는다. -> 웹 서버 처리 -> 코드 -> 유저 응답
- 톰캣
  - 한 개의 스레드가 한개의 요청을 끝까지 순서대로 처리
- Netty
  - 스레드는 이벤트를 처리하고 이벤트 루프로 돌아와 다른 이벤트가 있는지 확인
  - 이벤트 루프에서 이벤트를 끝없이 처리한다.
  - 물리적인 스레드와 같은 개수의 스레드만 사용한다
- 스레드는 cpu 코어에서 처리 된다.
  - 한정된 cpu 코어로 어떻게 처리? 
    - CPU는 한 개의 물리적인 스레드를 갖고 있다고 가정.
    - HW 영역 - 커널 영역(OS) - 유저 영역

# 8. Netty와 이벤트 루프 패턴
- 대부분 비동기 웹 서버 -> 이벤트 루프 사용하여 비동기 동작
  - ex) NIO Tomcat, Netty, ...
- 이벤트 루프란?
  - 스레드가 루프를 계속 돌면서 이벤트를 처리하는 방식
    - 이 루프를 도는 스레드를 이벤트 루프 스레드라고 한다.
- Netty와 이벤트 루프 동작 원리
  - OS는 이벤트가 완료될 때 마다 쌓아둔다.
  - 이벤트 루프 스레드는 1, 2를 계속 반복한다.
    1. 이벤트 루프에서 selector가 OS가 쌓아둔 이벤트가 있는지 조회
    2. 이벤트가 있다면 이벤트에 등록된 함수 실행

# 10. 가장 중요 - WebFlux에서는 블로킹 처리를 어떻게 해야 할까?
- WebClient를 사용하면 스케쥴러 스레드의 지원 없이도 완벽하게 비동기로 블로킹 회피가 가능하다.
- boundedElastic 사용한 스레드 분리 예시
  - 중요한 작업시 발생하는 모든 작업 -> 스레드 분리하여 JPA 활용
  - 실시간으로 저장할 필요 없는 데이터 -> MQ
  - 신뢰성이 떨어져도 되는 조회 -> R2DBC (비동기 DB 조회 라이브러리)
- WebFlux 블로킹 처리 원칙
  - R2DBC, WebClient 등을 이용해 I/O 블로킹을 가능한 최소화
  - 어쩔수 없이 블로킹 발생하는 요소, 병렬 처리, 오래 걸리는 작업(CPU 점유가 오래 걸리는) -> Scheduler를 이용해 스레드 분리
  - 리액티브 프로그래밍 기본 원칙은 스레드 변경 최소화 -> 불필요한 스레드 분리 삼가
